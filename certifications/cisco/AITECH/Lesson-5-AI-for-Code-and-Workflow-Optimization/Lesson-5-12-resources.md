# Lesson 5-12: Using AI for Secure Code Review

> Student follow-along resources, key concepts, and references for this sublesson.

## Overview

AI can **triage pull requests**—surfacing risky diffs, suggesting tests, and explaining unfamiliar code paths—but it cannot replace **policy, static analysis, secret scanning, and expert human judgment**. This sublesson places AI review in a **defense-in-depth** stack aligned with Lesson 3 (ethics/security) and Lesson 5-6 (quality tooling).

## Learning objectives

By the end of this sublesson you should be able to:

- Position AI review as early signal, not authoritative vulnerability verdict.
- Prompt with **threat model** context: exposure, data class, auth boundaries, frameworks.
- Integrate AI suggestions with SAST, dependency scanning, and mandatory human review.
- Document governance: which repos may use vendor AI review and how diffs are logged.

## Key concepts

### 1. Signal vs proof

| Layer | Provides | Does not replace |
| --- | --- | --- |
| AI review | Heuristics, coverage of boring paths | Formal verification |
| SAST / secrets scan | Tool-backed findings with rules | Business logic correctness |
| Tests | Regression signal | Adversarial security guarantees |
| Human security reviewer | Architecture-aware judgment | Automated speed |

### 2. Threat-aware prompting

Include: trust boundaries, sensitive data handled, authn/z expectations, deserialization or file upload surfaces, and known **banned APIs** for your org. Ask for **ranked findings** with severity rationale—then verify each.

### 3. Data handling

Ensure AI review vendors align with **data classification** for your code. Some teams restrict AI review to non-production mirrors or redacted branches.

## Why it matters / What's next

Secure SDLC already demands layered controls; AI simply adds another **fast, noisy** layer that needs tuning. **Lesson 5-13** zooms out to **additional tools** (Antigravity, OpenCode, Warp) and how to evaluate them without sprawl.

## Glossary

- **SAST** — Static application security testing; analyzes source without execution.
- **Threat model** — Structured description of assets, adversaries, and attack surfaces.
- **Triage** — Sorting findings by severity and likelihood for prioritized response.

## Quick self-check

1. Give an example of a finding you would trust more from SAST than from an LLM.
2. What belongs in a threat-model paragraph attached to a PR?
3. Why must customer PII influence whether AI review is allowed?
